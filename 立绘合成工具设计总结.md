# 立绘合成工具 设计总结

## 一、项目目标

编写一个带 GUI 的 Python 脚本，实现《千恋万花》角色立绘的可视化预览与无损 PNG 导出，替代手动 Photoshop 工作流。

---

## 二、核心需求

### 2.1 配置文件（config.ini）

- 格式：INI Section 模式，每个角色一个 `[section]`
- 只需指定 `.stand` 文件路径，其余文件均按 `.stand` 内 `filename` 字段在同目录下自动发现
- 示例：
  ```ini
  [芳乃]
  stand = ./芳乃/芳乃.stand

  [ムラサメ]
  stand = ./ムラサメ/ムラサメ.stand
  ```
- 工具启动时读取 config.ini，列出所有角色供选择

### 2.2 文件自动发现规则

`.stand` 文件内容示例（变体数量不固定，可能有 a / b / c / d 等）：

> **重要**：`a` / `b` 后缀代表角色的不同**视图**（如正面视图 / 侧面视图等），不是动作变体。具体哪个字母对应哪个视图不保证，但同一角色不同视图之间互相独立。
>
> 部分角色只有视图 `a`（无 `b`），由 `.stand` 文件中的 `filename` 列表决定。
>
> `_0` 后缀的文件**几乎一定**代表低分辨率版本（`a_0` = 视图 a 的低清版），但不能 100% 保证，当前以 `_0` 后缀作为低清判断依据。

```json
[
  { "filename": "芳乃a", "xoffset": 32, "yoffset": 180, "leveloffset": [...] },
  { "filename": "芳乃b", "xoffset": 42, "yoffset": 180, "leveloffset": [...] }
]
```

以 `.stand` 所在目录为基准，每个 `filename` 对应一整套文件：

| 用途 | 文件命名 | 示例（filename=芳乃a） |
|------|----------|------------------------|
| 高分辨率图层布局 | `{filename}.txt` | `芳乃a.txt` |
| 低分辨率图层布局 | `{filename}_0.txt` | `芳乃a_0.txt` |
| 资源索引 | `{filename}_info.txt` | `芳乃a_info.txt` |
| 高分辨率 PNG 图片 | `{filename}_{layer_id}.png` | `芳乃a_1857.png` |
| 低分辨率 PNG 图片 | `{filename}_0_{layer_id}.png` | `芳乃a_0_1857.png` |

> 每个 stand 变体（a / b）拥有**独立**的布局文件和图片资源。切换站位变体 = 切换到该变体对应的全套文件。

### 2.3 主界面 → 角色选择

- 启动后显示角色列表（从 config.ini 读取）
- 点击角色进入该角色的立绘编辑界面

### 2.4 角色编辑界面

用户可在此界面选择组合并实时预览：

| 选择项 | 说明 | 数据来源 |
|--------|------|----------|
| **站位变体** | 动态列出 `.stand` 中所有 `filename`（如 a / b / c …） | `.stand` 文件中的 `filename` 字段 |
| **分辨率** | 高清 / 低清 | 切换 `{filename}.txt` / `{filename}_0.txt` |
| **服装** | 从 info.txt 中提取去重后的 dress 名称列表 | `_info.txt` 中 `dress` 条目的第二列 |
| **表情** | 01~51，含 h（红晕）/ n（泪水）子变体 | `_info.txt` 中 `face` 条目的第二列 |
| **装饰物（独立勾选）** | 独立 CheckBox，可叠加于任意组合 | 见下方 |

**装饰物独立勾选列表：**
- `☐ ケモミミ（兽耳）` — 图层名含 `ケモミミ` 的 layer
- `☐ 頬（脸颊红晕）` — 固定图层 `頬`（layer_id 可能因变体而异，按名称匹配）
- `☐ 涙（泪水）` — `涙あり` 组下对应编号的泪水图层

### 2.5 预览区

- 实时显示当前选择组合的合成效果
- 画布自适应缩放（保持比例，适应窗口）
- 支持滚轮缩放、拖拽平移
- 背景使用灰白棋盘格表示透明区域

### 2.6 导出功能

- 按钮：「导出 PNG」
- 行为：
  1. 以选中服装的**全身图层**作为基准
  2. 将所有可见图层按其 `(left, top)` 坐标放置
  3. **自动扩展画布**：若有图层偏移为负值导致溢出，扩大画布（扩展区域为透明像素）
  4. 输出为无损 RGBA PNG
  5. 弹出保存对话框选择路径

## 三、数据映射关系

### 3.1 文件编码
- `.txt` 配置文件均为 **UTF-16-LE** 编码（BOM: `FF FE`）
- `.stand` 文件同理，使用自定义格式（`%[` 表示对象起始）

### 3.2 info.txt → 图层 txt → PNG 映射链

```
info.txt 资源名 ──(名称匹配)──> .txt 图层 name ──> layer_id ──> {filename}_{layer_id}.png
```

**服装映射：**
- `_info.txt`：`dress \t <服装名> \t diff \t <序号> \t <资源名>`
- 资源名直接匹配布局文件中图层的 `name` 字段
- **`diff` 字段**代表同一服装的不同动作/姿态变体（如 diff 1=基础姿态，diff 2=腕差分姿态）。同一服装的不同 diff 是**互斥**的——同一时刻只能激活一个 diff。
- 一个服装 + 一个 diff 由多个资源组成（如身体 + 前髪かぶせ等），但 **每个 diff 只对应一个身体图层**（不包含另一个 diff 的身体图层）。
- diff 数量可变：视图 `a` 通常有 2 个 diff，视图 `b` 通常只有 1 个 diff。但也有例外（单 diff 的 a 视图，或无 diff 的 b 视图）。

**表情映射：**
- `_info.txt`：`face \t <编号> \t base \t <资源路径>`
- 资源路径格式：`表情/笑顔1` 或 `表情（追加）/笑顔2b笑顔2e笑顔m`
- 取路径中 `/` 之后的部分作为图层名，匹配 `name` 字段
- 简单表情（01-23）：属于 `表情` 组
- 复合表情（24-51）：属于 `表情（追加）` 组

**h/n 后缀：**
- `h`：在基础表情上附加 `頬` 图层（blush）
- `n`：附加对应编号的泪水图层，如 `face 03n` → 资源 `涙あり/203 涙` → 图层名 `203 涙`

### 3.3 .stand 文件结构

自定义 JSON-like 格式，`%[` 表示对象起始：
```
[
%[
 filename:'芳乃a',
 xoffset:32,
 yoffset:180,
 leveloffset:[[0,-16],[0,-8],[0,-6],[0,-8],[0,-60],[0,-120]]
],
%[
 filename:'芳乃b',
 xoffset:42,
 yoffset:180,
 leveloffset:[[0,-16],[0,-8],[0,-6],[0,-8],[0,-60],[0,-120]]
],
]
```
- `filename`：文件前缀，用于派生所有配置文件和 PNG 路径
- `xoffset/yoffset`：全局屏幕偏移（导出时忽略）
- `leveloffset`：层级深度微调偏移（导出时忽略）

### 3.4 图层布局字段（每行）

| 字段 | 含义 | 示例 |
|------|------|------|
| layer_type | 0=图片层, 2=组容器 | 0 |
| name | 图层资源名 | 巫女服 |
| left | 左上 X 坐标 | 659 |
| top | 左上 Y 坐标 | 1508 |
| width | 图层宽度 | 1901 |
| height | 图层高度 | 3260 |
| type | 图片类型 | 13 |
| opacity | 不透明度(0-255) | 255 |
| visible | 默认可见性 | 0/1 |
| layer_id | 唯一标识=PNG文件名 | 1857 |
| group_layer_id | 所属组 | (可选) |

## 四、合成算法

### 4.1 基础合成
1. 创建透明画布（高分辨率 3600×5100 或低分辨率 1800×2550）
2. 将所有可见图层按各自的 `(left, top)` 放置
3. 图层叠放顺序：按在布局文件中出现的顺序（先出现=下层）

### 4.2 溢出处理（导出时）
- 计算所有可见图层的外接矩形（bounding box）
- 若 bounding box 超出画布边界（有负偏移或超出画布尺寸）：
  - 扩大画布至完整包含所有图层
  - 新增区域填充透明像素

## 八、已知问题 & 待修复

### 8.1 图层叠放顺序错误

**现象**：耳朵（ケモミミ）、表情、挂件等应叠在人物全身像**上面**，但目前被身体图层遮挡。

**原因分析**：

`芳乃a.txt` 中图层实际排列顺序（从前到后）：

| 文件位置 | 内容 | 应处于 Z 序 |
|----------|------|-------------|
| 行 3~7（开头） | 涙HL、頬、髪かぶせ、ケモミミ | **顶层**（最后绘制） |
| 行 9~21 | 涙图层（组 1963） | 中上层 |
| 行 22~48 | 复合表情（组 1937） | 中层 |
| 行 49~72 | 简单表情（组 1891） | 中下层 |
| 行 73~87（末尾） | 身体/服装（巫女服、裸等） | **底层**（最先绘制） |

即：**文件末尾 = 底层（先绘制），文件开头 = 顶层（后绘制）**。

当前代码按文件顺序依次 `alpha_composite`（文件开头先合成=底层，文件末尾后合成=顶层），导致身体在最上面，遮住了表情和耳朵。

**修复方案**：将图层合成顺序反转——文件末尾的图层先合成（底层），文件开头的图层后合成（顶层）。

### 8.2 表情列表显示不可读

**现象**：表情列表显示 `01`、`01h`、`02`、`02h` ……，用户无法直接看出每个编号对应的表情含义。

**原因**：`_info.txt` 中的 `face` 条目包含有意义的资源名：

```
face 01 base 表情/表情        → 主表情名 = "表情"
face 02 base 表情/笑顔1       → 主表情名 = "笑顔1"
face 03 base 表情/笑顔2       → 主表情名 = "笑顔2"
face 04 base 表情/きょとん     → 主表情名 = "きょとん"
face 01h base 表情/表情 + 頬   → 主表情名 = "表情" + 红晕
face 03n base 涙あり/203 涙    → 主表情名 = "203 涙"
face 24 base 表情（追加）/笑顔2b笑顔2e笑顔m  → 主表情名 = "笑顔2b笑顔2e笑顔m"
```

**修复方案**：从资源路径中提取主表情名称，格式为 `{编号} {主表情名}`，h/n 后缀附加标记。例如：

```
01 表情
01h 表情 +頬
02 笑顔1
03n 笑顔2 +涙
24 笑顔2b笑顔2e笑顔m
```

### 8.3 点击卡顿（约 1 秒）

**现象**：选择服装/表情后界面卡顿约 1 秒。

**原因**：每次选择变更都执行完整的合成管线——从磁盘加载所有 PNG、全分辨率（3600×5100）合成、生成棋盘格背景、缩放到预览尺寸。

**修复方案**（待前两个问题修复后实施）：

1. **缓存已加载的 PNG**：用 `dict[layer_id] = PIL.Image` 缓存，切换选择时只需切换可见性，无需重新加载
2. **惰性棋盘格生成**：用 Pillow 的 tile 或预渲染小方块后重复拼接，而非逐格绘制
3. **分离高清合成与预览合成**：预览使用低分辨率版本（若选中高清），仅导出时使用全分辨率

### 8.4 鼠标滚轮缩放不流畅

**现象**：缓慢逐格滚动可以缩放，连续快速滚动时无反应。

**原因**：每次滚轮事件触发 `_zoom_in`/`_zoom_out` → 立即调用 `_render_to_canvas`，后者包含棋盘格生成 + 图像缩放 + Canvas 重绘。连续滚动时事件密集涌入（Windows 下每次滚轮产生多个 `MouseWheel` 事件），前一次渲染未完成即被下一次打断，UI 事件循环被阻塞，表现为"无反应"。

**修复方案**：对滚轮事件做防抖处理——不立即执行 `_render_to_canvas`，而是用 `after(80ms)` 延迟合并渲染，密集事件只触发最后一次。

### 8.5 预览底图尺寸应所见即所得

**现象**：预览底图始终使用名义画布尺寸（高清 3600×5100 / 低清 1800×2550），导致预览区域有大量空白透明边距，不符合"所见即所得"。

**原因**：当前 `_composite` 使用 `final_w = max(canvas_w, max_x + offset_x)` 以名义画布尺寸为下限。

**期望行为**：

- 仅选中人物 → 底图 = 人物图层的包围盒（bounding box）
- 选中人物 + 表情，表情完全在人物范围内 → 底图 = 人物包围盒
- 选中人物 + 挂件，挂件超出人物范围 → 底图自动扩展以容纳挂件
- 即：**底图 = 所有可见图层的外接矩形**，不做无意义的名义尺寸填充

**修复方案**：`_composite` 中改为 `final_w = max_x - min_x`、`final_h = max_y - min_y`，所有图层坐标减去 `(min_x, min_y)` 偏移。仅当无可见图层时保留一个最小占位画布。

### 8.6 图层重叠顺序必须确定且不受用户点击顺序影响

**现象**：若用户先选表情再选人物，或先选人物再选表情，结果中的图层 Z 序必须一致。

**原因**：当前代码通过 `resource_names` 集合收集可见资源名，再按布局文件顺序遍历图层匹配，因此无论用户选择顺序如何，`visible_layers` 的排序始终由文件顺序决定。此逻辑已正确，但需结合 8.1 的反转修复确保 Z 序自上而下为：特效 → 耳朵 → 泪水 → 表情 → 身体（底层）。

**修复方案**：无需改动集合收集逻辑，只需确认 8.1 的反转合成在 `_composite` 中正确地按逆文件顺序合成。

### 8.7 点击已选中项应取消选中

**现象**：当前选中服装/表情后，再次点击同一项不会取消选中，用户无法回到"未选"状态。

**期望**：点击已选中项 → 取消选中（selection_clear），底图重新计算。

**修复方案**：将 Listbox 的 `<<ListboxSelect>>` 替换为 `<ButtonRelease-1>` + `after_idle` 延迟检测，比较新旧选中值——相同时取消选中，不同时切换选中。

## 十、已知问题 & 待修复（第二轮）

### 10.1 鼠标滚轮缩放卡顿

**现象**：8.4 的 80ms 防抖方案仍有卡顿感，连续快速滚动时响应不及时。

**原因分析**：

`_render_to_canvas` 每帧都在做：
1. `_make_checkered_bg` — 用 Python 双层循环逐 tile paste 生成全尺寸棋盘格（即使 tile 是 2×2 预渲染的，大画布仍有大量 paste 操作）
2. `Image.resize` — 全分辨率图像缩放到预览尺寸
3. `ImageTk.PhotoImage` — PIL → tkinter 转换

三者叠加导致每帧渲染耗时超过 80ms，防抖只是合并了事件，并未减少单帧开销。

**修复方案**（可多选组合）：

- **A. 预渲染棋盘格 tile 并用 Canvas create_image 平铺**：生成一个 tile PhotoImage，在 Canvas 上 tiling 显示，避免每次缩放重新生成棋盘格。
- **B. 增大防抖延迟**：将 80ms 提到 150ms，减少渲染帧数。
- **C. 缩放时使用低分辨率占位**：缩放过程中先用较小的缩略图渲染，缩放停止后再用全分辨率。即引入「草稿渲染」和「最终渲染」两级。
- **D. 后台线程合成**：在独立线程完成 `_composite` + 缩放 + 棋盘格合成，主线程只负责 `PhotoImage` 创建和 Canvas 绘制。

### 10.2 身体图层重叠问题（`diff` 维度缺失导致多动作混合）

**现象**（用户手动确认）：选中某套服装后，同一服装的不同「动作」（diff）的身体图层被同时叠加，导致预览中出现多只手等视觉错乱。

**根因分析**：

`_info.txt` 中每套服装按 `diff` 字段区分不同动作变体：

```
dress  巫女服  diff  1  巫女服          ← 动作1：基础身体
dress  巫女服  diff  1  髪かぶせ
dress  巫女服  diff  2  巫女服腕差分    ← 动作2：腕差分身体
dress  巫女服  diff  2  髪かぶせ
dress  舞服    diff  1  舞巫女服        ← 动作1
dress  舞服    diff  1  髪かぶせ
dress  舞服    diff  2  舞巫女服腕差分  ← 动作2
dress  舞服    diff  2  髪かぶせ
```

当前 `parse_info_file` **丢弃了 `diff` 字段**，将同一服装的所有 diff 的资源合并为一个集合：

```python
# 当前：忽略 diff，合并所有资源
dresses['巫女服'] = {'巫女服', '巫女服腕差分', '髪かぶせ'}
```

这导致选中「巫女服」时，diff 1 和 diff 2 的 3 个 body 图层（`巫女服`、`巫女服腕差分`、`髪かぶせ`）**同时被激活并叠加**。实际上这 3 个图层分别属于不同的动作姿态，是互斥关系——同一时刻只能显示**一个 diff** 的身体图层。

**数据结构纠正**：

`dress` 条目结构为：`dress \t <服装名> \t diff \t <序号> \t <资源名>`

解析应保留 diff 维度：
```
dresses[服装名][diff序号] = {资源名集合}
```

例如：
```
dresses['巫女服'][1] = {'巫女服', '髪かぶせ'}
dresses['巫女服'][2] = {'巫女服腕差分', '髪かぶせ'}
```

**修复方案**：

- **方案 A（推荐）**：修改 `parse_info_file` 保留 diff 结构，UI 中增加「动作」选择器（或嵌入服装列表为二级选择）。用户选择「巫女服 + diff 1」时仅激活 diff 1 对应的图层。
- **方案 B**：将 diff 展开为平级服装名，如「巫女服 (diff 1)」「巫女服 (diff 2)」直接列在服装 Listbox 中。

### 10.3 站位变体和分辨率应使用统一的选择控件

**现象**：当前「站位变体」和「分辨率」使用 `Radiobutton`，而「服装」和「表情」使用 `Listbox`，UI 风格不一致。

**期望**：全部使用 `Listbox`，统一样式和行为（包括 toggle 取消选中）。

**修复方案**：

- 将站位变体的 Radiobutton 替换为 Listbox（单选 + toggle），默认为第一个变体。
- 将分辨率的 Radiobutton 替换为 Listbox（单选，不允许取消选中，始终保留一个选中值），默认「高清」。
- 统一所有 Listbox 的 toggle 行为逻辑。

### 10.4 无选中内容时预览区应为空白

**现象**：打开角色编辑界面后，即使未选择任何服装/表情/装饰物，右侧预览区仍显示 `髪かぶせ` 图层（公共资源交集）。

**原因**：`_get_visible_layer_ids` 始终将所有 dress 的公共资源（`髪かぶせ`）加入可见集，即使 `selected_dress` 为空。

**期望**：没有任何选中项时，右侧预览区为空白（仅显示棋盘格背景）。

**修复方案**：

- 当 `selected_dress` 为空（无服装选中）时，`_get_visible_layer_ids` 返回空集。
- 仅当用户明确选择了一个服装后，才激活该服装的资源（含公共资源）。

### 10.5 涙（泪水）应归属于表情系统，而非独立装饰物

**现象**：当前「涙」是一个独立 CheckBox 装饰物，与「ケモミミ」「頬」并列。

**数据验证**：

`_info.txt` 中已有完整的 `n` 后缀 face 条目：

face 07n base 涙あり/107 涙

**结论**：涙是表情的一种子变体（`n` 后缀），不是可独立叠加的装饰物。游戏数据也明确将涙图层归入 `涙あり` 组（group_layer_id=1963），与特定表情编号绑定。

**修复方案**：

- 「頬」（脸颊红晕）同理——它已作为 `h` 后缀嵌入表情系统（如 `01h 表情 +頬`），也应从装饰物中移除，或保留但明确其来源（`h` 后缀自带 `頬` 图层）。

### 10.6 缺失文件时静默跳过，用户无感知

**现象**：若某个 layer 对应的 PNG 文件缺失，当前代码在 `_composite()` 中静默 `continue` 跳过，用户无从得知合成结果不完整。

**代码位置**：`sprite_composer.py` `_composite()` 方法：

```python
png_path = self._get_png_path(layer['layer_id'])
if png_path is None or not png_path.exists():
    continue  # 静默跳过
try:
    img = self._load_png(str(png_path))
except Exception:
    continue  # 异常也静默跳过
```

**期望**：在加载角色数据时（`load_variant_data` 或 `_load_data`）对所有需要的外部文件做一次存在性检查，若有缺失则弹出对话框列出缺失文件路径，让用户知晓资源不完整。

**修复方案**：

- 在 `_load_data` 加载完变体数据后，遍历所有 layer 检查对应 PNG / layout / info 文件是否存在。
- 收集缺失文件列表，若不为空则弹出 `messagebox.showwarning`，列出缺失文件路径。
- 可选：在状态栏持续显示「⚠ 缺失 N 个文件」。

### 14.1 缩放倍率手动输入

## 十六、config.ini 动态装饰物配置方案

### 16.1 设计原则

装饰物不应在代码中硬编码。所有角色专属装饰物及其服装绑定关系均通过 `config.ini` 配置，方便后期维护和添加新角色。

### 16.2 配置格式

在 `config.ini` 中为每个角色添加可选的 `decorations` 键，值为分号分隔的装饰物条目。每个条目格式：

```
装饰物显示名: 图层匹配关键词 [, bind: 服装关键词]
```

- **图层匹配关键词**：用于在布局文件中匹配 `name` 字段（支持子串匹配）
- **bind（可选）**：指定与此装饰物绑定的服装关键词。带 `bind` 的装饰物**不创建 CheckBox**——其资源已包含在对应 dress 条目中（见 `_info.txt`），选中匹配服装后自动出现，不选中则不出现。此类装饰物通常有多款样式与不同服装一一对应，二元开关无法表达

### 16.3 完整配置示例

```ini
[芳乃]
stand = ./所有角色资源/芳乃/芳乃.stand
decorations = 涙HL（泪水高光）: 涙HL; ケモミミ（兽耳）: ケモミミ, bind: 耳

[心子]
stand = ./所有角色资源/心子/心子.stand
decorations = 涙HL（泪水高光）: 涙HL

[小春]
stand = ./所有角色资源/小春/小春.stand
decorations = 汗: 汗; 涙: 涙; 頬線: 頬線

[ムラサメ]
stand = ./所有角色资源/ムラサメ/ムラサメ.stand
decorations = 息（呆れ）: 息（呆れ）

[レナ]
stand = ./所有角色资源/レナ/レナ.stand
decorations = 涙: 涙

[廉太郎]
stand = ./所有角色资源/廉太郎/廉太郎.stand
decorations = 涙: 涙

[比奈実]
stand = ./所有角色资源/比奈実/比奈実.stand
decorations = 涙: 涙

[芦花]
stand = ./所有角色资源/芦花/芦花.stand
decorations = キラン(ニヤリ2): キラン

[茉子]
stand = ./所有角色资源/茉子/茉子.stand
decorations = 涙: 涙
```

### 16.4 代码实现要点

1. **解析**：`config.ini` 的 `decorations` 键解析为列表，每项为 `(display_name, match_pattern, bind_dress_keyword)`
2. **动态 CheckBox**：`_build_ui()` 中根据 decorations 列表动态创建 CheckBox，去除当前硬编码的 `_has_kemonomimi` / `_has_hoho` 逻辑
3. **bind 装饰物不创建 CheckBox**：带 `bind:` 的装饰物**不创建独立 CheckBox**。原因：此类装饰物有多种款式与不同服装一一对应（如兽耳有 `ケモミミ(巫女服用)` / `ケモミミ(舞巫女服用)` / `ケモミミ` 三种），二元开关无法表达。它们随 `_info.txt` 中 dress 资源自动出现——选中带耳服装时 dress 资源集已包含正确的兽耳 PNG，无需额外勾选。**不带 bind 的装饰物**（如 `涙HL`、`汗`、`涙` 等）仍创建 CheckBox 供用户独立开关。
4. **图层匹配**：`_get_visible_layer_ids()` 中遍历**仅有 CheckBox 的装饰物**（即 `_deco_vars` 中存在的条目），按 `match_pattern` 匹配图层名。bind 装饰物跳过此循环——其资源完全由 dress 选择驱动。
5. **頬 保留特殊处理**：`頬` 是所有角色通用的装饰物（独立于上述配置），按名称直接匹配，保留现有逻辑

> **修正记录**（2026-05-11）：16.4 第3条「自动绑定」（选中服装后自动勾选+禁用 CheckBox）改为「bind 装饰物不创建 CheckBox」。原因是二元开关无法表达多款式兽耳与服装的一一对应关系，且此类装饰物本应随服装自动出现/消失。
